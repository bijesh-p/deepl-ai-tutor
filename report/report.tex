\begin{abstract}
AI Tutor turns an uploaded document into an interactive learning module. The
input can be a PDF, a slide deck, or a lecture transcript. A background
pipeline splits the document into topics. For each topic it writes an
explanation, a diagram, and comprehension questions. A separate adaptive tutor
is built as a LangGraph state machine. It asks the learner questions to gauge
their understanding. It tailors the explanation to that level and grades free
text answers. After a wrong answer it gives a hint or a simpler explanation.
The pipeline also produces a quiz, a mastery report, and a cohort analysis for
each module. This report describes the tutor architecture, the pipeline, and
three additions built on top of it. First, every language model call now passes
through a guardrail layer. The layer checks for prompt injection, output
safety, and topic relevance. It blocks a call with a clear message instead of
failing silently. Second, a knowledge graph is built for each module. It records
which concepts are prerequisites of others. The tutor can then ground a hint in
the right earlier concept rather than in text that only sounds similar. Third,
every generated question is tagged with one of six Bloom's taxonomy levels in
place of an informal difficulty label.
\end{abstract}

\end{frontmatter}

\section{Introduction}

A static document is hard to learn from on its own. A good learning version
needs explanations at the right depth, diagrams, checks for understanding, and
a tutor that adapts to what the learner already knows. Building this by hand
does not scale. AI Tutor automates it with two systems that work together. A
content generation pipeline turns an uploaded document into a sequence of
topics. A short chain of language model calls then writes each topic's
explanation, diagram, and questions. An adaptive tutor runs a loop over those
topics. It diagnoses what the learner already knows, presents each concept at
the right depth, asks a follow up question, and then advances or remediates
based on the answer. The tutor also stores its hints and explanations in a
retrieval store built from the same content~\cite{lewis2020retrieval}.

Section 2 describes the architecture: the content pipeline, the adaptive tutor,
and the tool servers and vector store they share. Sections 3 to 5 describe three
additions built on it: a guardrail layer on every language model call, a
knowledge graph that grounds a hint in a prerequisite concept, and a switch from
informal difficulty labels to Bloom's taxonomy levels. Section 6 reports
evaluation results and Section 7 outlines future work. The appendices give
supporting detail, including three reliability lessons that recurred at the
boundaries between these systems.

\section{System architecture}

\subsection{Ingestion pipeline}

A parser specific to the file type turns each upload into one shared model of
titled sections, so the rest of the system never sees the original format. A PDF
is read through PyMuPDF (one section per table of contents entry, else per page);
a slide deck becomes one section per slide; a Word file splits at each heading;
and a WebVTT transcript is split by its own classifier (Appendix A). Each parser
bounds how much it reads, so a long document cannot push unbounded text
downstream.

\subsection{Content pipeline}

A background thread splits an uploaded document into topics with a sliding
window: after each chunk, a language model call decides whether enough material
has built up to form one coherent topic, and a length limit forces a publish if
no window closes cleanly. For each topic, further calls write an explanation, a
diagram, and comprehension questions. If the diagram is not well formed with at
least one edge, a fallback call writes a bullet summary instead, so every topic
ends up with a diagram or a summary. Each topic's explanation is also narrated to
an mp3 with edge-tts, a local text to speech engine that needs no language model
call. Each topic is shown as soon as its own chain
finishes, so the learner can start before later topics complete. A final call
then writes the end of module quiz, and the module is saved to the database and
the vector store (Section 2.5).

\subsection{Adaptive tutor}

The adaptive tutor is a LangGraph state machine (Figure 1). A diagnostic call
generates multiple choice questions on the learner's prior knowledge; scoring
them is the only deterministic step and sets a presentation depth of beginner,
intermediate, or advanced that shapes every later call. To present a concept the
tutor reuses the pipeline's enriched content adapted to that depth, or retrieves
the closest matching content when the topic was not enriched yet (Section 4), as
happens after resuming a session. It then asks a follow up question and grades
the free text answer. A correct answer advances; an incorrect one goes to a
remediation step that gives a grounded hint or simplifies the explanation, then
loops back to ask again.

\subsection{Tool servers}

All effects outside the language model are isolated behind three tool servers
that follow the Model Context Protocol~\cite{anthropic2024mcp}, reached through a
single client. A document server parses an uploaded PDF, slide deck, Word file,
or transcript and extracts embedded images; any other input format would need a
new tool on that server. An assessment server checks that a piece of generated
output matches the schema the calling code expects. A storage server saves a
finished module to the database and reads and writes the vector store
(Section 2.5). Because every server is reached through the same client, the
pipeline and tutor never import a parsing library, database driver, or vector
store client directly. Appendix A gives the full pipeline diagram, the tools each
server exposes, and the VTT transcript path.

\subsection{Vector retrieval}

Right after a topic is enriched, its explanation is embedded by a small local
model~\cite{reimers2019sentence} and saved into a per module collection through
the storage server. This store is the system of record for each concept's text:
the knowledge graph in Section 4 decides which concept to fetch, but the text
always comes from here, never from the graph. The tutor queries it only when the
pipeline's own content for the current topic is missing, and every query is
filtered to the current module, for the reason given as the third reliability
lesson in Appendix E.


\subsection{Multi-provider access}

An LLM factory gives the model side the same uniformity the tool servers give the
data side: every call site is written against one interface. A per user setting,
changeable from the sidebar at runtime, routes calls to Anthropic, Portkey, or a
local Ollama model. The full path from upload to a finished session has been
checked end to end against all three, so switching providers changes only
configuration, not code.

\subsection{Evaluation and observability}

The system separates two kinds of evaluation. The first judges the learner and
drives the tutor: a call grades the free text answer and decides whether the
concept is mastered, and a deterministic step scores the diagnostic quiz. The
second judges the tutor's own output after a session, without affecting it: three
DeepEval metrics (answer relevancy, faithfulness, and clarity, each with a 0.5
pass threshold) run only when a user presses Run Evals, with the session's own
provider as judge. Every call also emits an OpenTelemetry trace to a local Arize
Phoenix server for inspection, with export to LangSmith available as an optional
feature enabled from the \texttt{.env} file. Section~\ref{sec:eval-results}
reports the results. Alongside these
runtime evaluations, the codebase carries a pytest suite covering both unit tests
of individual components, such as the parsers, the content pipeline, and the
guardrail wrapper, and end to end tests that exercise the full path from upload
to a finished session against each provider.

\section{Guardrails on every language model call}

The system makes language model calls from fourteen sites across the tutor and
pipeline, none of which checked their input or output. Every call now goes
through a wrapper that the LLM factory places around the provider client, so
every site is covered, including ones added later such as the knowledge graph's
relation extractor (Section 4). The wrapper runs four checks: a prompt injection
check~\cite{greshake2023not} and an output quality check use fixed patterns,
while a content safety check and a topic relevance check each make a second
provider call. The relevance check runs only when the caller passes the current
topic, which only the function grading a learner's free text answer does; the
other sites build prompts from text the system already generated. A failed check
raises a plain error that the existing error handling shows to the user, and a
failure in the second call does not block the session. Twenty six tests cover the
wrapper. Appendix B gives the full table and a regression the tests caught.

\section{Graph-guided retrieval}

The plain similarity search in Section 2.4 has no notion that one concept is a
prerequisite of another, so a hint could only re-explain the concept the learner
just missed, not the earlier one they actually lacked. The system now builds a
graph for each module~\cite{hogan2021knowledge} that records typed relationships
between concepts. It is built once every topic is enriched: a language model call
reads the module's concepts and returns the relationships it finds, and any
relationship to an unknown concept is dropped. If the call fails, the module
still gets a graph from the parts that need no model, such as document order.

When the tutor needs context, a retrieval function picks concepts relevant to the
current one and the kind of help needed (related concepts and prerequisites for a
hint, prerequisites further back for a simplification), then fetches each one's
text from the vector store in Section 2.5. It falls back to plain similarity
search when a module has no graph, the concept is unknown, or any step fails,
following the third reliability lesson in Appendix E. Appendix C gives the
relationship types and the retrieval algorithm.

\section{Bloom's taxonomy for question generation}

The system generates questions in four places: the end of module quiz, each
topic's comprehension questions, the pre lesson diagnostic, and the tutor's
follow up question. Each used to carry an informal easy, medium, or hard label;
every question now carries one of Bloom's six cognitive levels
instead~\cite{anderson2001taxonomy}: remember, understand, apply, analyze,
evaluate, and create. This follows Scaria et al.~\cite{scaria2024bloom}, who
found that a persona, reasoning before writing, and stating each level's
definition in the prompt worked best, while hand written examples made results
worse. Each place uses a different slice of the six levels: the quiz mixes all
six, comprehension questions use the first three, the diagnostic uses only
remember and understand, and the follow up question maps the learner's depth to a
pair of levels. The end of module quiz is timed, and on expiry it is submitted
automatically. Appendix D gives the full definitions and the question fields that
changed.

\section{Evaluation results}
\label{sec:eval-results}

The three end of session quality metrics are answer relevancy, faithfulness, and
clarity. Relevancy checks that the explanation addresses the question asked,
faithfulness that it says nothing the source does not support, and clarity
(defined for this project) that it is understandable on its own. Each passes at a
score of 0.5 or above. Scores are stored per metric and read back as a mean, a
pass rate, and a count, so a session with several questions gives one aggregated
result per metric rather than only the last answer's score.

As a worked example, we ran the three metrics on a module generated from a slide
deck on the Eyeriss deep learning accelerator~\cite{chen2016eyeriss}. All six
slides passed the 0.5 threshold on every metric, with an overall average of
0.799 (Table~\ref{tab:eyeriss-agg}); the lowest scores fell on the slide about
data movement as an energy bottleneck. Appendix F gives the per slide table and
the tutor steps that drive mastery during a session.

\section{Future work}

Two experimental features were prototyped on a separate GitHub branch but are
not yet part of the main system. The first is a chatbot that lets a student
interact with the tutor in free conversation rather than only through the
guided question and answer loop. The second is animation support that would
illustrate a concept with a short motion graphic alongside its static diagram.
Neither was stable enough to merge, so both are left for future work.

\section{Conclusion}

AI Tutor's content pipeline and adaptive tutor are each a short chain of language
model calls connected by retries, fallbacks, and a per call decision about
whether to ground the call in retrieved content. Neither is a single prompt, and
neither is a fully autonomous agent. The three additions, the guardrail layer,
the knowledge graph, and the Bloom's taxonomy labels, were each added at the
boundary of this chain rather than by changing it, which is why each could be
tested on its own. The reliability lessons point the same way: every recurring
failure sat at a boundary between systems, never inside the language model call
itself.

The regression mentioned in Section 3 happened as follows. The function that
grades a learner's answer was changed to pass the current topic into the
shared \texttt{generate} call, so the topic relevance check could run. A test
that builds a provider client directly, without going through the guardrail
wrapper, then failed with an error. The shared interface that every provider
client implements did not yet declare a topic argument; only the wrapper did.
The fix added that argument, with a default of no value, to the shared
interface and to every provider client. A provider client used on its own and a
provider client wrapped in guardrails now accept the same call. The test suite
caught this before the change was committed. The lesson echoes those in
Appendix E. A
change that looks safe because it only adds an argument can still break a caller
that does not go through the new code path.

The graph is stored as a directed graph that allows more than one relationship
between the same pair of concepts. It uses the NetworkX
library~\cite{hagberg2008exploring} and is written to a file per module in the
GraphML format. GraphML keeps the typed attributes on each relationship and can
be opened and read in a text editor for debugging.
A relationship of the prerequisite type must not form a cycle, since a cycle
would mean a concept is a prerequisite of itself once you follow the chain
around. After extraction, the system checks for cycles. It removes the lowest
confidence relationship in any cycle it finds, then records that it did so.

The retrieval function takes the module, the current concept if known, a
plain text query to fall back on, and which of the three kinds of help is
needed. It loads the module's graph. If none exists, it calls the same plain
similarity search the tutor always had and returns that result directly.
Otherwise it selects candidate concepts as described in Section 4 and fetches
each one's full text from the vector store. If the graph returned fewer
concepts than the function was asked for, it tops up the remainder with plain
similarity search on the original query. The selected text is then joined
together and returned in the same shape the tutor's calling code already
expected. The calling code therefore needed only a small change to use this
function in place of the plain search it called before.

\section{Bloom's taxonomy levels in full (supports Section 5)}

The six levels run from lowest to highest. Remember is recalling a fact or a
definition. Understand is explaining an idea in your own words. Apply is using a
concept in a new situation. Analyze is breaking a concept into its parts and
seeing how they relate. Evaluate is judging a concept against a standard. Create
is putting parts together into something new. The field that used to hold a
difficulty label on a question now holds one of these six level names. A quiz
that used to be generated at a single difficulty is now generated as a mix
across all six levels in one pass.

\section{Reliability lessons}

Three problems recurred across the pipeline and the tutor, each at a boundary
between systems rather than inside a language model call.

The vector store's default embedding function downloads an eighty megabyte
model the first time it runs. Before the fix, this download happened on the
exact request that was building a new user's first topic. The user saw nothing
beyond a normal wait. The storage tool server has the same kind of cost on its
own first call. Importing the numeric libraries it depends on can take anywhere
from about one second to over thirty seconds, depending on what else is running
on the machine. A call with no time limit would simply hang if that import took
unusually long. Both costs are now paid once, in a background thread, when the
application process starts, well before any real user reaches that code path.
The storage tool server's client now also enforces an explicit time limit on
every call as a second line of defense.

The diagram rendering failure had three distinct causes, found one after
another. First, a diagram with no usable content still produced a non empty
string. The code that prepares the diagram's source text always adds a required
header line, even to an empty diagram, so a check for an empty string did not
catch this case and the renderer attempted to draw nothing. Second, a page that
shows several topics as tabs keeps every tab's content in the page at once and
only hides the ones not currently selected. The library that measures text size
for layout returns a size of zero for anything inside a hidden tab, and the
rendering library only measures text once. A diagram first rendered while its
tab was hidden therefore stayed broken even after the tab was opened. Third, a
diagram with a cycle in it could make the rendering library's layout step run
for an unbounded amount of time with no result. Each of these three was fixed
the same way: a check before depending on the thing that check verifies, a
visibility check before measuring, and a time limit on the layout step that
falls back to the same bullet point summary if the limit is reached.

The retrieval scoping problem showed up as the tutor occasionally giving a
hint that read fine on its own but was clearly about a different topic than the
one the learner was struggling with. Retrieval calls were then changed to
filter by the current module and to run only when the pipeline's own content
for the current topic was missing. This stopped happening, because the search
space for any one call became much smaller and only ever contained content that
could plausibly be relevant.
